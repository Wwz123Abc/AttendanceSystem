# 考勤系统 — 后端实现与数据库说明文档

> 版本日期：2026-07-14　覆盖范围：后端实现逻辑、数据库表与关联关系、所有类与函数的功能说明。
> 本版相对 2026-06-26 版的核心变化：**原来按考勤组+类型配置的 `ApprovalFlow`（JSON 多级审批）机制已彻底移除**，改为"考勤组配审批人名单（`AttendanceGroupApprover`）+ 考勤组的审批层级（`ApprovalLevel`：一级/二级）"；新增出差申请类型、午间必打卡窗口、多打卡地点、员工扫码自助登记、找回密码、打卡时间表/员工信息表导出等功能；补全了此前完全缺失的 **Razor Pages 后台管理层** 说明（大量管理功能实际实现在 Pages/Admin、Pages/Report、Pages/Approval，不在 Controllers 里）。

## 一、项目概述

| 项 | 内容 |
|---|---|
| 类型 | 企业考勤管理系统（网页 + Web API + 钉钉对接） |
| 框架 | ASP.NET Core 8（Razor Pages + Controllers 混合，管理后台以 Razor Pages 为主） |
| ORM | EF Core + Pomelo MySQL（`ServerVersion.AutoDetect` 自动适配：生产 MySQL / 开发 MariaDB） |
| 认证 | Cookie 认证 + 基于角色的授权策略 |
| 日志 | Serilog（控制台 + 按天滚动文件） |
| 报表 | NPOI 导出 Excel |
| 命名约定 | 表名、列名全部 PascalCase（与 C# 类/属性一致） |

### 分层结构

```
Pages(Razor 页面) / Controllers(Web API)   ← 表现层
        ↓
Services(Interfaces + Implementations)     ← 业务层（核心逻辑）
        ↓
Data(AttendanceDbContext) + Models(Entities) ← 数据访问与领域模型
        ↓
MySQL / MariaDB
```
横切：Middlewares（当前用户上下文）、Helpers（Claims 工厂、Excel 导出）、BackgroundServices（定时任务）、Models/Options（强类型配置）。

**注意**：员工管理、考勤组管理（含审批人/审批层级配置）、部门管理、班次管理、假期管理、员工信息、管理看板、月度报表、提交申请、待我审批这几大后台功能，**实际实现在 Razor Pages 的 PageModel（OnGet/OnPost Handler）里，不是 Controllers**。`AdminController`/`ReportController` 这两个 API 控制器仍然存在，但功能明显更简陋（不支持黑名单、批量操作、扫码登记确认、审批人/打卡地点配置、模板报表导出等），大概率是早期遗留、未被当前页面实际使用，见第六节说明。

### 请求处理管线（Program.cs）

`UseStaticFiles → UseRouting → UseSession → UseAuthentication → UseAuthorization → CurrentUserMiddleware → MapRazorPages / MapControllers`。根路径 `/` 重定向到 `/Login`。仅**开发环境**启动时自动 `MigrateAsync()` 迁移并播种默认管理员（库里一个用户都没有才播种：总公司部门 + 总公司考勤组 + `admin/Admin@123`）；生产环境用幂等迁移脚本，不自动建表。

生产环境（现已扩展到开发环境同样生效）配有全局异常兜底：`InvalidOperationException→400`、`KeyNotFoundException→404`，其余 `→500`；API 请求返回 JSON，页面请求返回友好 HTML，避免裸 500。

### 授权策略

| 策略 | 允许角色 | 用途 |
|---|---|---|
| ManagePolicy | Admin、Clerk | 后台管理、报表、看板、钉钉同步 |
| ApprovePolicy | Admin、Clerk、Supervisor、TeamLeader | 审批处理 |

`/api` 路径请求未登录/无权限返回 401/403；页面请求未登录/无权限则重定向到登录页。

### 4.3 IApprovalService（审批，本次重点变化）

| 函数 | 功能 |
|---|---|
| SubmitApprovalAsync(applicantId, dto) | 提交申请：算请假/加班/出差时长 → 生成单号 → `CreateApprovalStepsAsync` 建节点 → 通知首个审批人 |
| HandleApprovalAsync(approverId, dto) | 审批处理：**逐层递进顺序闸**（前面还有更靠前的待办节点，禁止越级处理）；驳回→整单驳回+作废后续所有待办节点；通过且有下一节点→流转；通过且终审→整单通过并回写考勤；处理后通知申请人 |
| CancelApprovalAsync(userId, id) | 撤销（仅待审批可撤） |
| QueryApprovalsAsync(query) | 分页查询（多条件） |
| GetApprovalDetailAsync(id, requesterUserId, isManager) | 详情（含审批节点）；仅申请人本人 / 该单审批人 / 管理员可见，否则返回 null（防越权） |
| GetPendingForApproverAsync(approverId) | 待某人审批列表（只返回当前正轮到该人这一级的单） |
| GetMyApprovalsAsync(userId, status?) | 我提交的申请 |
| GenerateRequestNoAsync(type) | 单号：前缀（补卡BK/请假QJ/加班JB/**出差CC**）+ 日期 + 当日流水号 |
| GetAvailableApproversAsync(userId) | **（新）**查该用户所在考勤组配置的审批人名单，供提交页下拉选择 |
| CreateApprovalStepsAsync（私有，核心重写） | 见下方"审批人决定逻辑" |
| ResolveFallbackApproverAsync（私有） | 兜底审批人：无名单无上级时选一名在职管理员/文员（优先同考勤组，排除申请人本人） |
| NotifyApproversAsync/NotifyNextApproverAsync/NotifyApplicantAsync/AddNotificationAsync（私有） | 各环节站内通知 |

**审批人决定逻辑（`CreateApprovalStepsAsync`，取代已删除的 ApprovalFlow 机制）**：

1. **第一级**：考勤组配置了 `AttendanceGroupApprover` 审批人名单 → 员工必须自选其中一人（否则抛"请选择有效的审批人"）；名单为空 → 回退申请人的 `SupervisorUserId`（直属上级）→ 仍无 → 兜底指派一名在职管理员/文员。
2. **第二级（新逻辑，仅当考勤组 `ApprovalLevel == Level2` 时生成）**：第一级节点成功生成后，自动追加 `StepOrder=2` 节点，审批人固定是**申请人自己的 `SupervisorUserId`**，同样无上级则兜底管理员/文员。第一级没能生成节点时不会生成第二级，保证顺序完整性。

二级审批的语义是固定的"一级审批人（组内自选）通过后，自动追加申请人直属上级审批一次"，不是可以任意配置更多级的通用机制。

## 五、控制器层（Controllers）— API 路由

|---|---|---|

## 八、关键业务流程串联

**① 打卡→算考勤**：`PunchAsync` 入职/节假日/多地点定位校验 → 写流水 → 取/建日记录 → 按打卡类型（上班/**午间打卡**/下班）分支 → 统一工时公式算迟到/早退/工时/状态，支持夜班跨天续接。

**② 审批流转（本次重点变化）**：`SubmitApprovalAsync` 建单 + 建节点（一级=考勤组审批人名单自选/回退直属上级/兜底管理员；**若考勤组配了二级审批，一级通过后自动追加申请人直属上级作为二级**）+ 通知 → `HandleApprovalAsync` 逐级流转（顺序闸禁止越级）/ 驳回作废后续节点 / 终审回写考勤（补卡/加班/请假/出差四种类型）→ 全程站内通知。

**③ 月度汇总**：每月 1 日或手动 `GenerateMonthlySummaryAsync` 写 `MonthlyAttendanceSummary`，供报表1/2导出；"模板月度汇总表"和"打卡时间表"是独立路径，直接从每日记录现算，支持任意统计周期，不依赖这张汇总表。

**④ 钉钉数据流**：TokenProvider（缓存令牌）→ ImportEmployees（建部门/考勤组/员工）/ AutoMapByJobNumber（按工号映射 userid）/ Sync（拉打卡+请假落库，同步时当场算工时、做定位比对、回填午间打卡窗口）。触发：员工管理页按钮（手动）/ 后台定时（可选，默认关闭）。

## 九、配置与启动要点

- 连接串：`ConnectionStrings:Default`；生产由 `appsettings.Production.json` 覆盖。
- AppSettings/DingTalk/AMap 均已 `Configure<TOptions>` 强类型绑定并真实读取生效。
- DI 注册：核心业务服务 Scoped；钉钉 5 个客户端为 Typed HttpClient；`IPasswordResetService`/`IEmployeeRegistrationService` Scoped；内存缓存；后台 HostedService。
- 文件上传限制：`FormOptions.MultipartBodyLengthLimit` 约 50MB（对应最多 5 个 10MB 附件）。
- 异常处理：开发/生产环境均已启用全局兜底（此前只有生产环境启用）。
- 迁移：开发环境启动自动迁移；生产用幂等脚本 `dotnet ef migrations script --idempotent`。
