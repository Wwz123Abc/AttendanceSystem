# 考勤和审批管理系统 — 数据库表结构文档

## 文档说明

本文档详细描述考勤和审批管理系统的数据库表结构，包括 **15 个业务数据表**（另有 EF Core 自带的 `__EFMigrationsHistory` 迁移历史表，不计入业务表）及其字段定义、约束条件和表间关系。内容按当前代码库（实体类 + `dotnet ef migrations` 生成的真实表结构）核对生成，与实际数据库一致。

## 表概览

| 表名 | 说明 | 字段数 |
|---|---|---|
| User | 用户表（员工/管理员账号） | 25 |
| Department | 部门表 | 11 |
| AttendanceGroup | 考勤组表 | 11 |
| AttendanceGroupApprover | 考勤组审批人名单表 | 3 |
| AttendanceGroupLocation | 考勤组打卡地点表 | 6 |
| ShiftSchedule | 班次计划表 | 19 |
| ShiftAssignment | 班次分配表 | 6 |
| AttendancePunch | 打卡流水表 | 11 |
| AttendanceRecord | 考勤记录表 | 19 |
| MonthlyAttendanceSummary | 月度考勤汇总表 | 16 |
| ApprovalRequest | 审批请求表 | 23 |
| ApprovalStep | 审批节点表 | 8 |
| Holiday | 假日表 | 7 |
| Notification | 通知表 | 9 |
| EmployeeRegistration | 员工自助登记表 | 15 |

⚠️ **和历史版本文档的重要差异**：`ApprovalFlow`（含 `StepsConfig` JSON 多级审批配置）这张表**已经在 2026-07-07 的迁移里被彻底删除**，不要再参考任何提到它的旧文档。现在的审批人/审批层级配置机制是 `AttendanceGroupApprover`（审批人名单）+ `AttendanceGroup.ApprovalLevel`（一级/二级），详见下文。

## 1. User（用户表）

用途：存储所有员工/管理员账号信息，包括登录凭证、组织归属、联系方式、证件信息。

| 字段名 | 数据类型 | Null | Key | 说明 |
|---|---|---|---|---|
| Id | int | NO | PRI | 主键，自增 |
| EmployeeNo | varchar(50) | NO | UNI | 工号，唯一，作为登录账号 |
| RealName | varchar(50) | NO | | 真实姓名 |
| PasswordHash | varchar(256) | NO | | 密码哈希（PBKDF2-SHA256 + 16 字节盐 + 1 万次迭代，永不存明文） |
| DepartmentId | int | YES | MUL | 所属部门（外键 → Department.Id） |
| Position | varchar(100) | YES | | 岗位名称 |
| Role | int | NO | | 角色：1管理员/2文员/3主管/4班组长/5员工 |
| AttendanceGroupId | int | YES | MUL | 所属考勤组（外键 → AttendanceGroup.Id） |
| SupervisorUserId | int | YES | MUL | 直属上级（自引用 → User.Id）。审批流默认第一审批人来源，也是"二级审批"里第二级审批人的来源 |
| Phone | varchar(20) | YES | MUL | 手机号 |
| Email | varchar(100) | YES | | 邮箱 |
| AvatarUrl | varchar(500) | YES | | 头像图片地址 |
| HireDate | date | YES | | 入职日期 |
| IsActive | tinyint(1) | NO | | 是否在职（false=已停用，不能登录） |
| LastLoginAt | datetime(6) | YES | | 最后一次登录时间 |
| CreatedAt | datetime(6) | NO | | 创建时间 |
| UpdatedAt | datetime(6) | NO | | 最后修改时间 |
| DingTalkUserId | varchar(64) | YES | | 钉钉用户 userid（对接钉钉用，没绑定则为空） |
| IsBlacklisted | tinyint(1) | NO | | 是否在黑名单（永不录用）；拉黑时同时把 IsActive 置 false |
| IdNumber | varchar(18) | YES | MUL | 身份证号（18 位） |
| ContractCompany | varchar(100) | YES | | 合同公司（实际签合同/发工资的公司主体，可能与所在部门的公司不同） |
| EmergencyContactName | varchar(50) | YES | | 紧急联系人姓名 |
| EmergencyContactPhone | varchar(20) | YES | | 紧急联系人电话 |

| IdCardPhotoUrl | varchar(500) | YES | | 身份证照片访问地址 |

说明：`EmploymentTypeText`（正式工/临时工）、`Status`（在职/已停用/黑名单）是按 `Role`/`IsActive`/`IsBlacklisted` 实时推导的计算属性，**不落库**。

| 字段名 | 数据类型 | Null | Key | 说明 |
|---|---|---|---|---|

| DeptName | varchar(100) | NO | | 部门名称 |
| DeptCode | varchar(50) | YES | | 部门编码 |
| ParentId | int | YES | MUL | 上级部门（自引用 → Department.Id），顶级部门为空 |
| CompanyName | varchar(100) | YES | | 所属公司/分子公司 |

## 3. AttendanceGroup（考勤组表）

用途：考勤组配置——打卡规则（定位/午休晚餐扣时）+ 审批规则（审批人名单从属表 + 审批层级）。

| 字段名 | 数据类型 | Null | Key | 说明 |
|---|---|---|---|---|
| Id | int | NO | PRI | 主键 |
| GroupName | varchar(100) | NO | | 考勤组名称 |
| CompanyName | varchar(100) | YES | | 所属公司（展示用，由跟随部门的公司名自动派生，不手填） |
| ClerkUserId | int | YES | | 负责该组的文员用户编号 |
| EnableLocationPunch | tinyint(1) | NO | | 是否启用定位打卡 |
| LunchBreakMinutes | int | NO | | 午休时长（分钟），算工时时自动扣掉，默认 60 |
| DinnerBreakMinutes | int | NO | | 晚餐时长（分钟），算工时时自动扣掉，默认 30 |
| IsActive | tinyint(1) | NO | | 是否启用 |
| CreatedAt | datetime(6) | NO | | 创建时间 |
| UpdatedAt | datetime(6) | NO | | 最后修改时间 |
| **ApprovalLevel** | int | NO | | **（2026-07-14 新增）审批层级：1=一级审批（仅班组长）、2=二级审批（班组长通过后自动追加申请人直属上级审批一次）。默认 1** |

打卡地点不再是本表的单一经纬度字段，改成一对多的 `AttendanceGroupLocation`（见下）；审批人名单同样改成一对多的 `AttendanceGroupApprover`（见下）。

## 4. AttendanceGroupApprover（考勤组审批人名单表）

用途：考勤组 ↔ 审批人的多对多关联。一个考勤组可以配多个（地位平等的）审批人，员工提交申请时从名单里选一个人来审批第一级。**这张表 2026-07-07 取代了原来的 `ApprovalFlow`（JSON 多级配置）机制。**

| 字段名 | 数据类型 | Null | Key | 说明 |
|---|---|---|---|---|
| Id | int | NO | PRI | 主键 |
| AttendanceGroupId | int | NO | MUL | 属于哪个考勤组（外键 → AttendanceGroup.Id，级联删除） |
| UserId | int | NO | MUL | 审批人是谁（外键 → User.Id，`Restrict`——防止误删审批人导致历史审批节点丢失） |

唯一索引 `(AttendanceGroupId, UserId)` 防止同一个人被重复配进同一个组。管理页（考勤组管理）里，新增/修改时审批人的候选范围收窄为"班组长"角色，但存量已配置的非班组长审批人会继续保留生效。

## 5. AttendanceGroupLocation（考勤组打卡地点表）

用途：一个考勤组可以配多个打卡地点（如总部大楼、分部仓库各算一个），员工打卡落在其中任意一个的有效半径内即算定位通过。

| 字段名 | 数据类型 | Null | Key | 说明 |
|---|---|---|---|---|
| Id | int | NO | PRI | 主键 |
| AttendanceGroupId | int | NO | MUL | 属于哪个考勤组（外键，级联删除） |
| LocationName | varchar(200) | YES | | 地点名称（如"总部大楼"） |
| Latitude | double | NO | | 纬度 |
| Longitude | double | NO | | 经度 |
| RadiusMeters | int | NO | | 有效打卡半径（米），默认 500 |

## 6. ShiftSchedule（班次计划表）

用途：定义"一个班怎么上"——几点上下班、迟到早退容忍、跨天夜班、午间必打卡窗口等。

| 字段名 | 数据类型 | Null | Key | 说明 |
|---|---|---|---|---|
| Id | int | NO | PRI | 主键 |
| AttendanceGroupId | int | NO | MUL | 属于哪个考勤组 |
| ShiftName | varchar(50) | NO | | 班次名称（如正常班/早班/夜班） |
| ShiftType | int | NO | | 班次类型：1固定班/2弹性班/3自由班 |
| WorkStartTime | time(6) | NO | | 上班时间 |
| WorkEndTime | time(6) | NO | | 下班时间 |
| LateToleranceMinutes | int | NO | | 迟到容忍分钟数，默认 5 |
| EarlyLeaveToleranceMinutes | int | NO | | 早退容忍分钟数，默认 5 |
| EarliestClockInMinutes | int | NO | | 最多可提前多少分钟打上班卡，默认 60 |
| OvertimeThresholdMinutes | int | NO | | 加班已改为必须走审批，这个字段现在**不参与任何计算**，是历史遗留列 |
| IsCrossDay | tinyint(1) | NO | | 是否跨天班次（夜班，下班时间落到第二天） |
| StandardWorkHours | decimal(65,30) | NO | | 标准工时（小时），默认 8 |
| Color | varchar(20) | NO | | 日历上显示的颜色 |
| IsActive | tinyint(1) | NO | | 是否启用（软删除用） |
| CreatedAt | datetime(6) | NO | | 创建时间 |
| UpdatedAt | datetime(6) | NO | | 最后修改时间 |
| RestDaysOfWeek | varchar(20) | NO | | 每周休息日，逗号分隔的星期几编号（0=周日...6=周六），默认空字符串 |
| MidCheckStartTime | time(6) | YES | | **（新增）午间必打卡窗口开始时间**，和 MidCheckEndTime 要么都配、要么都空 |
| MidCheckEndTime | time(6) | YES | | **（新增）午间必打卡窗口结束时间**。窗口内没有任何打卡时，当天工时只算下午 |

## 12. ApprovalStep（审批节点表）

用途：记录每一张申请单每一级审批人是谁、审了没、什么意见，支持多级审批按顺序流转。

| 字段名 | 数据类型 | Null | Key | 说明 |
|---|---|---|---|---|
| Id | int | NO | PRI | 主键 |
| ApprovalRequestId | int | NO | MUL | 属于哪张申请单（外键，级联删除） |
| ApproverUserId | int | NO | MUL | 这一级的审批人是谁（外键，`Restrict`） |
| StepOrder | int | NO | | 步骤序号，1=第一级、2=第二级，按顺序往后审 |
| ApprovalStatus | int | NO | | 这一级的处理结果 |
| Comment | varchar(1000) | YES | | 审批意见（驳回时必填） |
| HandledAt | datetime(6) | YES | | 处理时间（还没处理为空） |
| CreatedAt | datetime(6) | NO | | 创建时间 |

现在一张申请单最多生成两条节点：`StepOrder=1`（考勤组配置的审批人名单里员工自选的人，或退回直属上级/兜底管理员）；如果所属考勤组配置了"二级审批"，一级通过后自动追加 `StepOrder=2`（申请人自己的直属上级，或同样兜底）。逐层递进由服务层保证：前一级还没审完，后一级看不到也审批不了。

## 13. Holiday（假日表）

用途：法定节假日 / 公司休息日 / 调班补班日，决定某天要不要上班、算不算应出勤。

| 字段名 | 数据类型 | Null | Key | 说明 |
|---|---|---|---|---|
| Id | int | NO | PRI | 主键 |
| HolidayName | varchar(100) | NO | | 假期名称（如"国庆节"） |
| HolidayDate | date | NO | MUL | 具体哪一天 |
| HolidayType | int | NO | | 类型：1法定/2公司休息/3调班补班 |
| AttendanceGroupId | int | YES | MUL | 适用范围：空=全公司，有值=只对该考勤组生效 |

要点：
- User 是中心实体，几乎所有业务表都以 UserId 关联，且多为 Cascade（删人连带删其考勤/审批/排班/通知记录）。
- 审批人相关外键（`AttendanceGroupApprover.UserId`、`ApprovalStep.ApproverUserId`）统一用 `Restrict`：防止删除一个当过审批人的账号导致历史审批数据出现悬空引用。
- 部门、考勤组、上级对 User 用 SetNull：删除组织架构节点不会连带删掉员工记录。
- 关键唯一约束保证业务一致性：一人一天一条考勤记录/排班、一人一月一条月度汇总、审批单号唯一、考勤组审批人名单去重、员工工号唯一。

## 命名约定
